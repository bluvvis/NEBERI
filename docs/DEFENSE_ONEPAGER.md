# Одна страница: сценарий защиты NeBeri

Цель: за **10–12 минут** показать **продукт**, **инженерию** и **границы** без ухода в «мы заменили антифрод МТС».

## 1. Зачем продукт (30 с)

- Поток событий связи → **скор + объяснение** (`reasons[]` с `references[]` на открытые источники, `policy_version`) для интеграций BSS/SOC.
- Учебный **MVP**, вдохновлённый телеком-практикой; см. [`PRODUCT_SCOPE.md`](PRODUCT_SCOPE.md).

## 2. Живая демо (4–5 мин)

1. **Compose / Helm URL** — UI открыт.
2. **Симуляция** — пакет демо-сценариев; показать **разные** `risk_level` и **деталь** с правилами.
3. **Фильтр риска** — лента и блок метрик (полная выборка vs фильтр).
4. **OpenAPI** `/docs` — контракт, `idempotency_key`, схема ответа.
5. **Очистка** `POST /v1/events/purge` — сброс демо-данных.
6. **`/metrics`** — ingestion, high-risk, **`neberi_rule_fires_total`** по правилам (связь с [`OPERATIONS.md`](OPERATIONS.md), [`observability.md`](observability.md)).

## 3. Архитектура (2–3 мин)

- **FastAPI + Postgres**; скоринг **YAML** (версия в каждом ответе).
- **React** консоль; nginx в Compose проксирует `/v1`.
- **CI** — тесты API + web, образы, ручной deploy Helm.

## 4. Честные ограничения (1 мин)

- Нет реальных CDR/BSS; burst и ночное окно — **демо-логика** на своей БД.
- Скор — **сумма весов правил**, не калиброванный ML; см. [`INDUSTRY_SIGNALS.md`](INDUSTRY_SIGNALS.md) и [`analytics/OFFLINE_EVALUATION.md`](analytics/OFFLINE_EVALUATION.md) — как развивать без долга в hot path.

## 5. Вопросы, к которым готовы

- Почему правила, а не LLM на каждый запрос? — **latency, ПДн, воспроизводимость**.
- Как усилить скоринг? — **больше независимых признаков**, затем **offline ML** / калибровка, см. OFFLINE_EVALUATION.
- Как смотреть инциденты в проде? — **метрики + логи request_id**; дальше Grafana/алерты из `observability.md`.
