# Техническое задание (краткое) — NeBeri Fraud Signal API

**Версия:** 0.2 · **Статус:** черновик под критерии курса (аналитика / ТЗ).

## 1. Назначение

Сервис приёма **событий связи** (звонок, SMS, текст после ASR), расчёта **риска мошенничества** и возврата **структурированного объяснения** (сработавшие правила) для интеграции в продукты оператора.

## 2. Границы MVP

| Входит | Не входит (v1) |
|--------|----------------|
| REST `POST/GET /v1/events` | Реальные CDR из прод BSS |
| PostgreSQL, YAML-политика | Обучаемые ML-модели |
| Маскирование MSISDN в ответах | Персональные данные в логах |
| OpenAPI `/docs`, `/metrics` | SSO / OAuth для UI |
| Идемпотентность `idempotency_key` | Распределённый rate limit (Redis) |
| Rate limit на POST (slowapi, по IP) | WAF уровня периметра |

## 3. Функциональные требования

1. **POST /v1/events** — валидация тела, скоринг, сохранение, ответ `EventOut`.
2. Повтор POST с тем же **`idempotency_key`** — **HTTP 200**, тот же ресурс, без дубля в метрике ingestion.
3. **GET /v1/events** — пагинация `limit`/`offset`, фильтр `risk_level`.
4. **GET /v1/events/{id}** — деталь.
5. Политика скоринга — файл YAML + версия в ответе; ошибка конфигурации — **HTTP 500** с текстом `detail`.

## 4. Нефункциональные требования

- Доступность для демо: single-region.
- Логи: `request_id`, метод, путь, статус, длительность.
- Метрики: Prometheus на `/metrics`.
- Rate limit: настраивается `RATE_LIMIT_EVENTS` (например `120/minute`), отключается `RATE_LIMIT_ENABLED=false`.

## 5. Интеграция

- Клиент: JSON, UTF-8.
- Рекомендуется заголовок `X-Request-ID` для сквозной трассировки (задел под трейсы).

## 6. Приёмочные критерии (связка с ПМИ)

- Юнит-тесты: политика YAML, идемпотентность, API smoke.
- Subprocess-тест: rate limit **429** при лимите `2/minute`.
