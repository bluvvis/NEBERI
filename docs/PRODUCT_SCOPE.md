# Границы продукта и допущения (NeBeri Fraud Signal)

Документ для защиты и онбординга: **что это**, **что это не претендует быть**, **какие допущения** приняты в MVP за ~4 месяца.

## Что входит в scope MVP

- Приём событий **`call` / `sms` / `voice_text`** по REST (`POST /v1/events`).
- Скоринг по **версионируемой YAML-политике** с массивом **`reasons[]`** и при необходимости **`reasons[].references[]`** (аудит и объяснимость).
- **Маскирование** MSISDN в ответах, **идемпотентность** по `idempotency_key`, **rate limit** на ingestion.
- **OpenAPI** (`/docs`), **метрики Prometheus** (`/metrics`), **структурированные access-логи** с `X-Request-ID`.
- Веб-консоль: лента, фильтр по риску, деталь, демо-симуляция, очистка демо-данных (`POST /v1/events/purge`).
- Деплой: **Docker Compose**, **Helm chart**, **GitLab CI** (тесты, сборка образов).

## Что сознательно вне scope (non-goals)

- Замена **промышленного** антифрода оператора (BSS/OSS, реальные CDR, блокировки, юридические сценарии блока).
- **Онлайн-LLM** на каждый ивент (стоимость, задержка, ПДн, воспроизводимость на защите).
- Обучение и **сервинг production ML-модели** в hot path без отдельного MLOps-контура и датасета под NDA.
- Интеграция с реальными **IMEI / сотовыми вышками / DPI** — нет доступа к данным курса.
- Полная **i18n** UI и мультитенантность.

## Допущения данных

- Демо-тексты и сценарии **согласованы** с открытыми описаниями smishing/vishing и типовыми правилами; см. [`INDUSTRY_SIGNALS.md`](INDUSTRY_SIGNALS.md).
- Burst-метрики считаются по **уже сохранённым** событиям в БД демо-окружения; при маскировании номеров возможны **коллизии** для редких хвостов — зафиксировано в `PROJECT_PLAN.md` §7.

## ПДн и этика

- В ответах API — только **маскированные** номера и **укороченный** фрагмент текста (`text_excerpt`).
- Проект позиционируется как **учебный контур**; для продакшена потребуются политики хранения, согласия, DPIA и т.д. — не раскрываются в MVP.
