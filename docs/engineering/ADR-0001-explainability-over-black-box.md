# ADR-0001: Объяснимость и версия политики важнее «чёрного ящика»

**Статус:** принято  
**Дата:** 2026-04-22  
**Контекст:** учебный проект NeBeri, срок ~4 месяца, защита перед комиссией и потенциальным работодателем.

## Контекст

Нужен скоринг по событиям связи. Варианты: (A) только ML/LLM, (B) только статические правила, (C) правила + прозрачный путь к offline ML.

## Решение

Выбран подход **(C) с акцентом на (B) в runtime**:

- В ответе API всегда есть **`reasons[]`** (в т.ч. опциональные **`references[]`** из YAML — связь с открытыми источниками) и **`policy_version`**.
- Для `sms` / `voice_text` допускается **лёгкий ML-слой** (TF-IDF + LR, joblib) с полями **`rule_score`**, **`ml_fraud_proba`**, **`ml_model_version`** и весом **`ML_BLEND_WEIGHT`** — объяснимость сохраняется за счёт явного разделения «правила vs модель».
- Пороговые уровни и веса задаются в **YAML**, который можно менять без перекомпиляции кода (кроме смены схемы правил в коде — тогда нужен PR в `scoring.py`).

## Последствия

- **Плюс:** объяснимость, воспроизводимость на защите, простые тесты на политику.
- **Минус:** скор не является «вероятностью мошенничества» до калибровки и доп. признаков.
- **Отклонённые альтернативы:** онлайн-LLM на каждый `POST` (стоимость, ПДн, нестабильность); полноценный MLOps в репозитории курса без датасета.
